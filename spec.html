Abstract

The Visual Notation for OWL Ontologies (VOWL) defines a visual language for the user-oriented representation of ontologies. It provides graphical depictions for elements of the Web Ontology Language (OWL) that are combined to a force-directed layout visualizing the ontology.

This specification focuses on the visualization of the ontology schema (i.e. the classes, properties and datatypes, sometimes called TBox), while it also includes recommendations on how to depict individuals and data values (the ABox). Familiarity with OWL and other Semantic Web technologies is required to understand this specification.

Overview [back to top]

VOWL, the Visual Notation for OWL Ontologies, is based on only a handful of graphical primitives forming the alphabet of the visual language: Classes are depicted as circles that are connected by lines and arrows representing the property relations. Property labels and datatypes are shown in rectangles. Information on individuals and data values is either displayed in the visualization itself or in another part of the user interface.

The second ingredient of VOWL is a color scheme complementing the graphical primitives. It defines colors for the visual elements to allow for an easy distinction of different types of classes and properties. Although the color scheme is defined in an abstract way leaving room for customization, a set of concrete color recommendations provides guidance for developers.

Finally, the visual elements are combined to a graph that is rendered in a force-directed layout and that represents the ontology. VOWL defines splitting rules to specify which elements are multiplied and thus displayed several times in the visualization to improve the overall layout.

VOWL can be used with any version of OWL (i.e. OWL 1 and OWL 2), although not all language constructs are included in this specification. Best results may be achieved with OWL DL or the more restrictive profiles OWL 2 EL/QL/RL, as well as OWL Lite. VOWL also provides graphical depictions for elements of RDF and RDFS that are reused in OWL, such as rdf:Property or rdfs:subClassOf.

Basic Building Blocks [back to top]

This section introduces the graphical primitives and color scheme that form the basic building blocks of VOWL. It also defines splitting rules for nodes in the graph visualization and gives some general guidelines for using the notation.

Graphical Primitives

Table 1 lists all graphical primitives used in VOWL.

Table 1. Graphical primitives.
Name Primitive Description Application
Circle

Each circle represents a class of the ontology. The color of the circle depends on the type of class. There is an implementation-specific default size for circles.

If an ontology contains individuals and unless purely the schema (i.e. the TBox) is considered, the circle can become larger than the default size – in that case, its size corresponds to the number of individuals that are instances of the class. VOWL does not specify a particular function for mapping the number of individuals to the circle size – good results may be achieved with a logarithmic or square-root scaling in most cases. A maximum size may be imposed. Individuals inferred from subclasses are counted as well if reasoning is available and not deactivated.

The circles representing the special classes owl:Thing and rdfs:Resource are exceptions. They are not scaled but have a fixed size that is smaller than the default size of classes. In addition, the circles of these two classes have a dashed border and are multiplied in the visualization according to the rules described in Table 3.

owl:Class, rdfs:Class, owl:DeprecatedClass, rdfs:Resource, owl:Thing
Line

Lines are used to represent property relations. They usually have an arrowhead. The line style and arrowhead to use are determined by the type of the property (see below). If the same two classes are connected by multiple lines (i.e. if there are multi-edges between classes), the lines may be curved.

 rdfs:domain, rdfs:range, rdfs:subClassOf, owl:disjointWith, owl:intersectionOf, owl:unionOf, owl:complementOf
Arrowhead

Arrowheads indicate the direction of property relations by pointing to the object of the property – i.e., a datatype, class, or class instance that is defined as the range of that property.

Arrowheads representing range axioms are filled with the foreground color; arrowheads representing subclass relations are filled with the neutral color.

rdfs:range (with foreground color), rdfs:subClassOf (with neutral color)
Rectangle

Property labels and datatypes are shown in rectangles. If representing datatypes, the rectangles have the datatype color and a border. If representing property labels, they are borderless and colored according to the property type.

rdfs:Datatype, rdfs:Literal, property labels
Line style

The lines and borders of some class and property types are dashed or dotted. A dashed line indicates set operators and class disjointness (if visualized). A dashed border indicates special types of classes as well as literals. A dotted line is reserved for subclass relations.

dotted line: rdfs:subClassOf;
dashed line: owl:intersectionOf, owl:unionOf, owl:ComplementOf, owl:disjointWith;
dashed border: owl:Thing, rdfs:Resource, rdfs:Literal, and anonymous classes
Text

Text is used for all kinds of labels, cardinalities and other text strings, numbers, and symbols defined in ontologies. Moreover, a textbox with some information from the ontology header, such as the title and description of the ontology, may be displayed in the visualization.

rdfs:label, type of property ("Subclass of" for rdfs:subClassOf), additional type information in brackets (e.g. "deprecated", "functional", etc.), symbols in schematic illustrations (e.g. union symbol ("∪") for owl:unionOf)

Color Scheme

Table 2 lists the colors used in VOWL. They are defined in an abstract way, along with a description and a recommendation for a concrete color and its hex code. Developers may choose different colors, as long as they comply with the abstract descriptions. All colors are defined relative to the canvas color, which is assumed to be very light. The colors (and color descriptions) may be 'inverted' if the visualization is presented on a dark background.

While the colors ease the reading of the visualization, they are not required to understand it. The visualization is also understandable if printed in black and white or read by color-blind people, though some details may not be visible any longer (e.g. if a class is of type owl:Class or rdfs:Class). If these details should also be available in black and white prints of the visualization, they may be added as text (e.g. in the class label).

There may be conflicts in using this color scheme, e.g. when a deprecated class belongs to an external ontology. They should be resolved by giving one of the colors a higher priority and adding all information in the label and/or another part of the user interface (e.g. the sidebar). Splitting Rules

In some cases, elements of an ontology are connected to many other elements in the ontology. By displaying each of these relations, a large number of lines would point to the node representing such an ontology element, possibly resulting in a large number of crossing graph edges. Table 3 defines a number of rules that allow splitting up a node, so a single ontology element (such as a class) is represented by several nodes. Each relation linked to such an ontology element is then displayed only for one of the representations.

Table 3. Splitting rules.
Level Description Application
0 no splitting

The node appears is displayed only once. Any edge connected to the node is linked to the one graphical representation.

classes
1 splitting by class

The node appears once for every class it is connected to. If the node is linked to two classes, two representations of the node will be displayed. If the node is linked to one class by more than one property, all of those properties will be displayed with separate edges, but linking the class to the same graphical representation of the node.

owl:Thing, rdfs:Resource
2 splitting by property

The node appears once for every property it is used by. If the node is linked to two properties, irrespective of whether the other end of the edges is connected to the same class, two separate representations of the node will be displayed.

rdfs:Datatype, rdfs:Literal

General Guidelines

Some general guidelines and recommendations for usage and implementation are listed in the following:

Labels

If elements do not have an rdfs:label that can be depicted, it is recommended to take the last part of the URI as label, i.e. the part that follows the last slash (/) or hash (#) character. Labels may be abbreviated if they do not fit in the available space (e.g. "Winnie-the-Pooh" → "Winnie…"). The full label can be shown on demand (e.g. in a tooltip or sidebar).

Graph Layout

The visual elements are combined to a graph that should be rendered in an appealing and readable layout. The position of circles should be roughly determined by the number of inbound and outbound property relations, i.e. the more property relations a class has, the more centric its circle in the layout. It is recommended to use a force-directed graph layout (though other layouts may be equally or even more appropriate in certain cases). The users should be enabled to adapt the graph layout to their purposes.

Missing Domain and/or Range Information

If no domain and/or range for a property is specified, owl:Thing is used as domain and/or range. An exception are datatype properties without a specified range, where rdfs:Literal is used as range instead.

Visual Represention of owl:Thing

The visual representation of owl:Thing should only be used if a property has no domain and/or range, or if owl:Thing is explicitly specified as domain and/or range. It is not recommended to visualize subclass relations of owl:Thing, as every OWL class is implicitly a subclass of owl:Thing and the subclass relations would unnecessarily clutter the visualization.

External Elements

Basically, all elements whose base URI differs from that of the visualized ontology may be visually marked as external (this feature is optional). Alternatively, developers may define several colors to visually group external elements by namespace, as long as they adhere to the 'inverted coloring' of external elements (i.e. dark color with light font).

Highlighting

VOWL defines three modes of interactive highlighting, two of which are direct:

  1. Highlight on hover: changes the fill color of the hovered element (plus the color of the line and arrow if any) to the highlighting color. This highlighting mode is dependent on the environment (e.g., it cannot be usefully applied in many touch environments).
  2. Highlight on selection: changes the border color of the selected element (plus the color of the line and arrow if any) to the highlighting color. Note that it should also be possible to deselect elements that have been selected.

The third highlighting mode is indirect. It expresses some kind of relationship between elements, without connecting them visually by a line. It is used in VOWL to dynamically indicate sub- and superproperties of a hovered property. Like the highlight on hover, it changes the fill color of the element (plus the color of the line and arrow if any), but this time to the indirect highlighting color.

Conflicts

The deprecated color has priority over the external color, i.e. if a class or property from an external ontology is deprecated, it will be visualized in the deprecated color. In such cases, the text in the element will display all attributes in a comma-separated form (e.g. "deprecated, external"). If a property is symmetric and functional, the text will indicate "symmetric, functional" accordingly.

The equivalent class notation generally overrides other attributes. If a class in the visualized ontology is equivalent to a deprecated or an external class, it is displayed using the normal equivalent class notation.

Table 6. Graphical representation of further properties
Ontology Element Graphical Representation Description
owl:cardinality,
owl:minCardinality, owl:maxCardinality

Exact, minimal, and/or maximal cardinality constraints are written as numbers near the end of the property arrow. The representation is similar to the UML notation for cardinalities, i.e. the letters "x" and "y" in the example represent the minimum and maximum cardinality for that property. Exact cardinalities are represented by only one number, while the asterisk (*) denotes an unrestricted cardinality. Examples are "0..1" for a cardinality of 0 to 1, "1..*" for a minimum cardinality of 1, or "3" for a cardinality of exactly 3.

The default cardinality of most properties is any number (i.e. zero or more) and would be represented by a sole asterisk. However, it is recommended to only visualize cardinalities that are explicitely specified with owl:cardinality, owl:minCardinality and owl:maxCardinality. Default property cardinalities (incl. the global cardinaility constraints of functional and inverse functional properties) are better not shown in the visualization to avoid confusion and reduce visual clutter.

The color of the rectangle depends on the type of property, i.e. general color = owl:ObjectProperty, datatype property color = owl:DatatypeProperty, and rdf color = rdf:Property.

owl:disjointWith

Dashed line without arrowhead, connecting the disjoint classes and having the symbol of two disjoint sets as a label. The label has the general color and indication "disjoint" in brackets. If several classes are pairwise disjoint, this would be visualized by connecting each two of them. As an alternative to showing this element in the visualization, owl:disjointWith statements may be presented as details on demand in another part of the user interface, which is recommended as it usually results in a less cluttered graph.
rdfs:subClassOf

Dotted line starting at the subclass, having an arrowhead filled with the neutral color that points to the superclass. The representation is similar to the UML notation for generalizations but with a dashed line instead of a solid one to make it clearly distinguishable from other property relations. In addition, it is recommended to add the label "Subclass of" to the representation to clarify the semantics of the relation. The label has either a transparant background or is shown in a rectangle that has the neutral color (unless highlighted), which is recommended.
rdfs:subPropertyOf

If a property is hovered, its sub- and superproperties (if any) are dynamically highlighted using the indirect highligthing color and mode. The color of the rectangle depends on the type of property, i.e. general color = owl:ObjectProperty, datatype property color = owl:DatatypeProperty, and rdf color = rdf:Property.

owl:inverseOf

Object property representation with arrowheads at both ends, annotated with the label of the property and its inverse counterpart (interactive highlighting of direction). In order to emphasize which label denotes which direction in non-interactive contexts, each label may be slightly displaced toward its respective arrowhead (not shown in the example).
owl:equivalentProperty

Labels of equivalent properties are listed in squared brackets.
owl:equivalentClass

Class notation with double border (with the space colored in the neutral color). The main label is from a class with the same base URI (if there is any). The labels of equivalent classes (from external ontologies or from the same base URI) are added as a comma-separated list to the label in squared brackets.

This notation may also be used to represent owl:sameAs in OWL Full, where it can be defined between two classes.

Note that:

  • The visual representations of rdf:Property can always be instantiated by its more specific subclasses owl:ObjectProperty and owl:DatatypeProperty, as both types are subclasses of rdf:Property. The same applies to properties with specific attributes, such as functional properties [ref].

Note that the background of this property can have different colors. It is shown in the rdf color in this specification, as owl:FunctionalProperty is a subclass of rdf:Property and can therefore be applied to any property in the ontology (i.e. same notation with different background color for owl:ObjectProperty and owl:DatatypeProperty).

Note that:

  • Equivalent classes and properties a label of one of the equivalent elements is selected as the main label. If any of the elements are from the ontology being visualized (rather than external [ref]), a label from such an element will be preferred.

Data values

Elements such as owl:allValuesFrom,owl:someValuesFrom, owl:hasValue, rdfs:comment, rdfs:seeAlso, rdfs:isDefinedBy, owl:sameAs, and owl:DataRange (rdfs:datatype in OWL 2) can only be displayed with interactive events (either in a side panel or when hovering a certain class/individual).

Elements such as owl:Ontology, owl:differentFrom, owl:AllDifferent, owl:distinctMembers, owl:Restriction, owl:onProperty, owl:AnnotationProperty and owl:OntologyProperty serve as containers for other elements or define constraints for populating the ontology. They are not graphically represented by the visual notation.

Change Notes [back to top]

VOWL 1 distinguished between three layers that may be visualized --> ABox and TBox Version 1.0 of the VOWL specification distinguished between three layers that may be visualized: Conceptual Layer, Instance Layer and Integrated Layer (more information on the layers of the visual notation can be found in Negru & Lohmann 2013.), therefore in the current version (Version 2.0) we address the issues of on-demand interactive exploration and scalability of Visual Notation for OWL ontologies -- issues we identified in a comparative user study conducted on the VOWL Version 1.0 and UML representation for RDFS/OWL Ontology Definition Metamodel (ODM) (more details regarding the user study are available in Negru, Haag, and Lohmann 2013.).
