Abstract

The Visual Notation for OWL Ontologies (VOWL) defines a visual language for the user-oriented representation of ontologies. It provides graphical depictions for elements of the Web Ontology Language (OWL) that are combined to a force-directed graph layout visualizing the ontology.

This specification focuses on the visualization of the ontology schema (i.e. the classes, properties and datatypes, sometimes called TBox), while it also includes recommendations on how to depict individuals and data values (the ABox). Familiarity with OWL and other Semantic Web technologies is required to understand this specification.

Overview [back to top]

VOWL is based on only a handful of graphical primitives forming the alphabet of the visual language: Classes are depicted as circles (Table 4) that are connected by lines and arrows representing the property relations (Table 5). Property labels and datatypes are shown in rectangles (Table 6, Table 7 and Table 8). Information on individuals and data values is either displayed in the visualization itself or in another part of the user interface (Table 9).

The second ingredient of VOWL is a color scheme complementing the graphical primitives. It defines colors for the visual elements to allow an easy distinction of different types of classes and properties. Although the color scheme is defined in an abstract way leaving room for customization, concrete color recommendations provide guidance for developers.

Finally, the visual elements are combined to a graph that is rendered in a force-directed layout and that represents the ontology. VOWL defines splitting rules to specify which elements are multiplied in the visualization in order to make them less prominent.

VOWL can be used with any version of OWL (i.e. OWL 1 and OWL 2), though not all language constructs are included in this specification. Best results may be achieved with OWL DL or the more restrictive profiles OWL 2 EL/QL/RL, as well as OWL Lite. VOWL also provides graphical depictions for elements of RDF and RDFS that are reused in OWL, such as rdf:Property or rdfs:subClassOf.

Basic Building Blocks [back to top]

This section introduces the graphical primitives and color scheme that are the basic building blocks of VOWL. It also defines splitting rules for the graph visualization and gives some general recommendations for using the notation.

Graphical Primitives

Table 1. Graphical primitives.
Name Primitive Description Application
Circle

Each circle represents a class of the ontology. If an ontology contains individuals, the size of each circle corresponds to the numbers of individuals its class contains. VOWL does not specify a particular scaling method for the circle sizes - good results may be achieved with a logarithmic or square-root scaling in most cases. The color of a circle depends on the type of class it represents. Individuals inferred from subclasses are also counted if reasoning is available and not deactivated. If the ontology is not populated or if purely the schema (i.e. the TBox) is considered, all classes have the same default size.

An exception are the special classes owl:Thing and rdfs:Resource that are not scaled but have a fixed size that is smaller than the default size of the other classes. In addition, these two classes have a dashed border and are multiplied in the visualization according to the rules described in Table 3.

owl:Class, rdfs:Class, owl:DeprecatedClass, rdfs:Resource, owl:Thing, owl:Nothing
Line Lines are used for properties and set operators. When representing a property, the line has an arrow head (expect for the built-in properties owl:disjointWith and owl:equivalentClass), when representing a set operator, it has a dashed line style. The built-in property owl:disjointWith also has a dashed line style, while subclass properties have a dotted line style. If the same two classes are connected by multiple lines (i.e. if there are multi-edges between classes), the lines are curved. rdfs:domain, rdfs:range, rdfs:subClassOf, owl:disjointWith, owl:intersectionOf, owl:unionOf, owl:complementOf
Arrowhead Arrowheads indicate the direction of properties by pointing to the class or datatype being the range of the property. If multiple ranges are defined for a property, this is represented with an anonymous class and the notation of owl:intersectionOf in the visualization. Furthermore, arrowheads are used to indicate the direction of subclass relations. Arrowheads representing range axioms are filled with the foreground color (usually black); arrowheads representing subclass relations are filled with the neutral color (usually white). rdfs:range (with foreground color), rdfs:subClassOf (with neutral color)
Rectangle Property labels and datatypes are shown in rectangles. If used for datatypes, the rectangles have the datatype color and a border. If used for property labels, they are borderless and colored according to the property type (Table 5). rdfs:Datatype, rdfs:Literal, property labels
Line style

The lines and borders of some class and property types are dashed or dotted. A dashed line indicates the application of set operators to classes. It can also indicate disjoint classes (if this property relation is visualized at all). A dashed border indicates special types of classes (owl:Thing, rdfs:Resource, and anonymous classes, as used with set operators) and properties (rdfs:Literal). A dotted line is reserved for subclass relations.

 dotted line: rdfs:subClassOf; dashed line: owl:intersectionOf, owl:unionOf, owl:ComplementOf, and owl:disjointWith; dashed border: owl:Thing, rdfs:Resource), rdfs:Literal, and anonymous classes
Text Text is used for all kinds of labels defined in ontologies. If no label is given for an element, the last part of the URI is taken, i.e. the string that follows the last slash (/) or hash (#) character. Some property or class type Finally, a textbox with information from the ontology header, such as the title and description of the ontology, may be displayed in the visualization. rdfs:label, type of property ("Subclass" for rdfs:subClassOf), additional type information in brackets (e.g. deprecated, equivalent, etc.), symbol in schematic illustrations (e.g. union symbol ("∪") for owl:unionOf)

Color Scheme

Table 2 lists the colors used in VOWL. They are defined in an abstract way, along with a description and a recommendation for a concrete color and its hex code. Developers may choose different colors, as long as they comply with the abstract descriptions. All colors are defined relative to the canvas color which is assumed to be very light. The colors (and color descriptions) may be 'inverted' if the visualization is presented on a dark background.

While the colors ease the reading of the visualization, they are not required to understand it. The visualization remains understandable if printed in black and white or read by color-blind people, though some details may not be visible any longer (e.g. if a class is of type owl:Class or rdfs:Class). If these details should also be available in black and white prints of the ontology visualization, they may be added as text (e.g. in the class label).

There may be conflicts in using this color scheme, e.g. when a deprecated class belongs to an external ontology. Such cases are listed and resolved in the general recommendations.

Abstract color name Concrete color recommendation Description Application
Canvas #fff white Very light color with a good contrast to all other colors. canvas (background of the visualization)
Neutral #fff white Light color with a good contrast to all other colors (usually similar or identical to the canvas color). owl:Thing, arrowhead of rdfs:subClassOf, double border of owl:equivalentClass
Foreground #000 black Very dark color with a good contrast to all other colors, except the external color. text, lines, borders, symbol signs, arrowhead of rdfs:range
General #acf light blue Light color with a good contrast to the foreground color. owl:Class, owl:ObjectProperty (incl. subclasses), owl:disjointWith
Rdf #c9c light purple Color similar to the general color with a good contrast to the foreground color rdfs:Class, rdfs:Resource, rdf:Property (incl. subclasses)
Deprecated #ccc light gray Color that is clearly different from the other colors and which calls little attention. owl:DeprecatedClass, owl:DeprecatedProperty
External #36c dark blue Dark version of the general color, with the text in the neutral instead of the foreground color. Classes and properties from external ontologies that are linked by the ontology that is visualized.

Indirect Highlighting #f90 orange Color similar to the highlighting color.

Splitting Rules

Table 3. Splitting rules.
Level Description Application
0 no splitting The node appears is displayed only once. Any edge connected to the node is linked to the one graphical representation. Classes
1 splitting by class The node appears once for every class it is connected to. If the node is linked to two classes, two representations of the node will be displayed. If the node is linked to one class by more than one property, all of those properties will be displayed with separate edges, but linking the class to the same graphical representation of the node. owl:Thing, rdfs:Resource
2 splitting by property The node appears once for every property it is used by. If the node is linked to two properties, irrespective of whether the other end of the edges is connected to the same class, two separate representations of the node will be displayed. rdfs:Datatype, rdfs:Literal

General Recommendations

Some of the general guidelines and instructions / recommendations of usage and implementation are listed below:

Labels

If classes, properties, or individuals / elements do not have an rdfs:label, it is recommended to take the last / a part of the URI as label, i.e. the part after / the string / characters that follow the last slash (/) or hash (#) character / sign. Labels may be abbreviated if they are too long (e.g. Hieroglyphics → Hierogly…). The full label can be shown on demand (e.g. via tooltips) and/or in another part of the user interface (e.g. a sidebar).

Graph Layout

The graph layout should arrange the elements in an appealing and readable layout. The position of circles should be roughly determined by the number of inbound and outbound property relations, i.e. the more property relations a class has, the more centric the circle in the graph. Force-directed layouts / drawing algorithms / Algorithms for force-directed graph drawing / may be used for this purpose. They can be tailored to the visualized ontology by increasing or decreasing the forces between the classes.

Circle Sizes

The size is established by the number of individuals a certain class encompasses (if there are any). The class representation of owl:Thing has a fixed size, as it usually does not carry any particular domain information. It is recommended to use multiple representations of owl:Thing to avoid giving this element an importance in the visualization that it does not deserve.

Special Cases

- no domain or range: 1. object property --> owl:Thing (per class) 2. datatype property --> rdfs:Literal (each a single instance) Every class can have its own visual instance of owl:Thing, i.e. all properties that have this class as their domain (or range) but do not define an explicit range (or domain) are connected to the same visual instance of owl:Thing. In addition, there is one more visual instance of owl:Thing that is linked to all properties that do neither define domain nor range axioms or explicitly link to owl:Thing. Alternatively, this extra graph may consist of two or more visual instances of owl:Thing -- see Table 3 for information regarding splitting/multiplication of VOWL nodes; [ToDo: Example]

We do not recommend to visualize subclass relations of owl:Thing, as every OWL class is implicitly a subclass of owl:Thing [ref], so that the subclass relations would unnecessarily clutter the visualization. Instead, the visual representation of owl:Thing should only be used if a property has no domain or range or explicitly uses owl:Thing as its domain or range. The same holds for rdfs:Resource.

External ontologies

The elements can either be imported with owl:imports or are simply linked in the / via domain and range axioms. Basically, all elements that have a different base URI than the visualized ontology may be visually marked as external (this feature is optional). Alternatively, developers may define several colors to visually group external elements by namespace, as long as they do not use a color from the scheme and keep the 'inverted style' with the light font color.

Highlighting

VOWL distinguishes two/three modes of direct highlighting:

  1. Highlight on hover: changes the fill color of the element (plus the color of the line and arrow if it has any)
  2. Highlight on selection: changes the border color (plus the color of the line and arrow if it has any)
This is an interactive feature that is dependent from the environment (e.g. the first mode may not make sense in some touch environments). Note that it should also be possible to deselect highlighted elements. Indirect highlighting can be used to indicate a certain type of relationship between elements. It is used in VOWL to dynamically indicate sub- and superproperties of a selected / hovered property. Like the direct highlight on hover, it changes the fill color of the element (plus the color of the line and arrow if it has any) / is changed and is dependent on the interactive environment.

Interactive Representation of the VOWL notation is based on the following events for manipulating OWL ontology elements:

  • Hover = highlighting details about certain elements e.g. inverse properties;
  • Click = details on demand about the ontology elements;
  • Drag and drop = (re)arranging the elements;
  • Other interactions include: zoom, panning.

Conflicts / Special cases:

- deprecated class from an external ontology? --> color of deprecated, subtext deprecated and external
- combination of property types, e.g. symmetric property that is also functional, add all property specifics as a comma-separated list in the subtext (abbreviate according to the abbreviation rules)
- external class that is defined as equivalent class --> external class notation

Particular aspects regarding the graphical representation, which need to be noted:

  • Combined properties are represented by combining their elements, i.e. a transitive with a functional property [ToDo: Example]
  • The visual representations of rdf:Property can always be instantiated by its more specific subclasses owl:ObjectProperty and owl:DatatypeProperty are both subclasses of rdf:Property.
  • Some visual elements are "stronger" than others, i.e. imported class notation is used for both owl:Class and rdfs:Class, while deprecated class notation is used for all three (owl:Class, rdfs:Class, and imported class). The same holds for the graphical representation of deprecated properties that is visually stronger than the other properties (though the labels remain the same in all cases, pure or combined).

owl:cardinality,
owl:minCardinality, owl:maxCardinality

Exact, minimal, and/or maximal cardinality constraints are written near the end of the property arrow that the cardinality refers to. The notation is similar to UML, i.e. the letters "x" and "y" in the example represent the minimum and maximum cardinality (for the range). Exact cardinalities are represented by only one number, while the asterisk (*) denotes an unrestricted cardinality. As unrestricted is the default cardinality / arbitrary number (zero or more) of values, the asterisk is optional and usually be omitted / not shown.

Examples are "0..1" for a cardinality of 0 to 1, "1..*" for a minimum cardinality of 1, or "3" for a cardinality of exactly 3. Note that the cardinality always refers only to the range of the property axioms, so inverse properties are needed to define cardinality in both directions. The cardinality notation should not be used to depict the global cardinality of functional properties to avoid confusions.

owl:disjointWith Dashed line/arc with symbol of two disjoint sets as label. Alternatively, owl:disjointWith statements may be presented as details on demand (e.g. in the sidebar), which is recommended as it results in a less cluttered graph.
rdfs:subClassOf Dotted line/arc with a white arrow head at the end pointing to the superclass (arrow head like in UML but with a dashed line), with a white or transparent label background.
rdfs:subPropertyOf Sub-/Superproperties are indirectly / dynamically highlighted on hover.
owl:inverseOf Line/arc with arrow heads at both ends and annotated with the label of the property and its inverse counterpart (interactive highlighting of direction). In order to emphasize which label denotes which direction in non-interactive contexts, each label may be slightly displaced toward its respective arrowhead.
owl:equivalentProperty Labels of equivalent properties are listed in squared brackets.
owl:equivalentClass

Class notation with double border (with space in neutral color). The main label is from a class with the same base URI (if there is any). The labels of equivalent classes (from external ontologies or from the same base URI) are added as comma-separated list to the label in square brackets. They are abbreviated according to the rules above.

- main label = label of main class, i.e. - the only class containing / with the base URI (if there is any) - one of the classes with the base URI (if there are more) - one of the classes (if there is no class with a base URI)

This notation may also be used to represent owl:sameAs in OWL Full where it can be defined between two classes.

Individuals may be represented as sections in the circles. Individuals inferred from subclasses are usually also counted (if reasoning is available and not deactivated). Data values

For elements such as: owl:allValuesFrom, owl:someValuesFrom, owl:hasValue, rdfs:comment, rdfs:seeAlso, rdfs:isDefinedBy, owl:sameAs, and owl:DataRange (rdfs:datatype in OWL 2) interactive events must be used in order to be displayed (either in a side panel or when hovering a certain class/individual).

Elements such as owl:Ontology, owl:differentFrom, owl:AllDifferent, owl:distinctMembers, owl:Restriction, owl:onProperty, owl:AnnotationProperty and owl:OntologyProperty serve as containers for other elements or define constraints for populating the ontology. They are not graphically represented by the visual notation.

Change Notes [back to top]

VOWL 1 distinguished between three layers that may be visualized --> ABox and TBox Version 1.0 of the VOWL specification distinguished between three layers that may be visualized: Conceptual Layer, Instance Layer and Integrated Layer (more information on the layers of the visual notation can be found in Negru & Lohmann 2013.), therefore in the current version (Version 2.0) we address the issues of on-demand interactive exploration and scalability of Visual Notation for OWL ontologies -- issues we identified in a comparative user study conducted on the VOWL Version 1.0 and UML representation for RDFS/OWL Ontology Definition Metamodel (ODM) (more details regarding the user study are available in Negru, Haag, and Lohmann 2013.).
